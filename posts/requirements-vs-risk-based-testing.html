Why Requirements-Based Testing Fails in an AI World

Your spec document was written before the attacker showed up.

There is a foundational tension in software security testing that my current coursework keeps circling back to. On one side: requirements-based testing, which asks "does the software do what we said it would?" On the other: risk-based testing, which asks "how could an attacker break this — including ways we never thought of?"

For most of software development history, requirements-based testing was sufficient as a starting point. Threats evolved slowly enough that a well-written spec and a disciplined test plan could catch most of what mattered.

That era is over.

The Problem With Requirements

Requirements are written at a point in time. By the time software reaches deployment, that point in time may be months or years in the past. The threat landscape doesn't wait. New attack methodologies emerge. Cloud configurations shift attack surfaces. Third-party components introduce vulnerabilities that weren't in scope when the original architecture was designed.

Ransome, Misra, and Merkow (2023) summarize the shortcomings directly: design assumptions may be obsolete by the time software is ready for testing, and the original architecture may not account for future versions of third-party components not planned for in the initial implementation.

This is not a theoretical concern. It describes the operational reality of every software team shipping in 2026.

The Grok Case: When Language Is the Attack Surface

A recent incident involving Grok illustrates exactly why risk-based thinking is now non-negotiable. A threat actor manipulated the model using Morse code — encoding malicious instructions in a format the model's content filters weren't trained to recognize as a threat. The model processed the encoded instructions and was leveraged in a financial attack.

No requirements document would have included: "the system shall not respond to Morse code encoded attack instructions." That attack vector didn't exist as a known pattern when the system was built. Requirements-based testing would have passed this system with confidence. Risk-based testing — which assumes a creative adversary actively probing for unexpected inputs — has at least a chance of catching it.

This is the defining challenge of AI security. In traditional software, a vulnerability is a flaw in the code. You find it, you patch it, it's fixed. In AI systems, the vulnerability is often the model's core capability — its ability to process and respond to language. You cannot patch language understanding without breaking the product.

Prompt Injection: The Attack That Frameworks Aren't Ready For

What the Grok incident describes is a form of prompt injection — manipulating an AI system by crafting inputs that cause it to interpret malicious instructions as legitimate ones. OWASP has recognized this as the number one risk in their LLM Top 10, published specifically because traditional application security frameworks don't adequately address AI threat surfaces.

The guardrails that exist on large language models are trained on natural language threat patterns. Encoding instructions in Morse code, Base64, or other non-standard formats can bypass filters that weren't designed to decode those formats before evaluating content. The attacker doesn't need to find a code vulnerability. They need to find a way to make the model misinterpret context.

Risk-Based Testing as the Answer

The principle that software security testing takes the perspective that the tester is the attacker is not new — it predates AI by decades. What is new is the scope of creative adversarial inputs that now need to be considered.

Risk-based security testing builds test scenarios around misuse and abuse cases, not just intended use cases. It asks: what assumptions did we make about how this system would be used, and what happens when an attacker deliberately invalidates those assumptions?

For AI systems specifically, that question needs to include: what happens when the attacker communicates in a way the system wasn't trained to recognize as adversarial?

Key Takeaway

Requirements tell you what the software should do. Risk-based testing asks what an attacker could make it do instead. In a world where AI systems can be manipulated through language itself, the gap between those two questions has never been wider.
