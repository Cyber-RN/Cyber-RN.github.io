Why Requirements-Based Testing Fails in an AI World

Your spec document was written before the attacker showed up.

My current coursework keeps coming back to the same tension in software security testing. Requirements-based testing asks "does the software do what we said it would?" Risk-based testing asks "how could an attacker break this, including ways we never thought of?" For a long time, the first question was enough. Threats moved slowly enough that a solid spec and a disciplined test plan caught most of what mattered.

That's not the world we're in anymore.

The Problem With Requirements

Requirements are written at a specific point in time. By deployment, that moment could be months or years in the past. Threats don't wait. New attack methods show up. Cloud configurations shift the attack surface. Third-party components introduce vulnerabilities nobody planned for in the original architecture.

Ransome, Misra, and Merkow (2023) put it plainly: design assumptions may be obsolete by the time software is ready for testing, and the original architecture may not account for future versions of third-party components that weren't part of the initial plan.

This isn't a theoretical problem. It's happening on every software team shipping right now.

The Grok Case: When Language Is the Attack Surface

A recent incident involving Grok is a good example of why risk-based thinking matters. A threat actor manipulated the model using Morse code, encoding malicious instructions in a format the content filters weren't built to catch. The model processed them anyway and was used in a financial attack.

No requirements document would have said "the system shall not respond to Morse code encoded attack instructions." That attack didn't exist as a known pattern when the system was built. Requirements-based testing would have passed it clean. Risk-based testing, which assumes a creative attacker actively looking for unexpected inputs, at least has a shot at catching something like this.

That's the core challenge with AI security. In traditional software, a vulnerability is a flaw in code. You find it, patch it, done. In AI systems, the vulnerability is often the model's core capability: its ability to process and respond to language. You can't patch that without breaking the product.

Prompt Injection: The Attack Frameworks Aren't Ready For

What happened with Grok is a form of prompt injection, where an attacker crafts inputs that cause the model to interpret malicious instructions as legitimate ones. OWASP put this at number one in their LLM Top 10, published specifically because traditional application security frameworks don't cover AI threat surfaces well enough.

The guardrails on large language models are trained on natural language threat patterns. Morse code, Base64, and other non-standard encodings can slip past filters that were never designed to decode them before checking for threats. The attacker doesn't need a code vulnerability. They just need to find a way to make the model misread the situation.

Risk-Based Testing as the Answer

The idea that the security tester should think like an attacker isn't new. It predates AI by decades. What's new is the range of creative adversarial inputs that now have to be considered.

Risk-based testing builds scenarios around misuse and abuse cases, not just intended use. It asks: what assumptions did we make about how this system would be used, and what happens when an attacker deliberately violates those assumptions?

For AI systems, that question has to go one step further: what happens when the attacker communicates in a way the system wasn't trained to recognize as a threat at all?

Key Takeaway

Requirements tell you what the software should do. Risk-based testing asks what an attacker could make it do instead. When AI systems can be manipulated through language itself, the gap between those two questions keeps getting wider.
